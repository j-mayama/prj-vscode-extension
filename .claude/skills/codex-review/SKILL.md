# codex-review — Codex 独立レビュー & 修正ループ

Claude が書いたコードを、別の AI（OpenAI Codex）に非対話モードでレビューさせる。
P0/P1/P2は必ず修正し、P3は対象外として修正・持ち越しをせず、再レビュー後に
レビュー対象を自動コミットして、元のブランチへmergeする。pushはしない。

```
実装指示 → worktreeへ分離（実装を伴う場合）→ 実装 → Stop hook
  → 対象を確認 → codex exec review 実行 → 結果テキストを Claude が構造化
  → 指摘 0 件 …… 完了報告して終了
  → 指摘あり
       → P0/P1/P2 …… 必須修正
       → P3 …………… 対象外として結果にだけ記録
       → 離席中の判断 … AskUserQuestionを使わず進める
       → 対応分を修正 → 再レビュー → 先頭に戻る
  → 収束または最大往復回数 → 状態確定 → 自動コミット
      → 元のブランチへmerge → 報告
```

## Step -1: 実装前にworktreeへ移動する

**このリポジトリで`.codex-review-auto`が有効な場合、ファイルを変更する前に
このセッション専用のworktreeへ移動する。** レビューより前、実装より前の話であり、
`/codex-review`を呼ばない普通の実装依頼にも適用される。

### なぜ

1つのチェックアウトを複数セッションで共有すると、**このスキルの有無に関係なく壊れる**。
gitの差分は「そのツリー全体」でしかないため、セッションAのレビューにはセッションBの
書きかけコードが必ず混ざり、Aの必須修正がBのコードに当たる。さらに両者が同じファイルを
編集すれば、後から書いた側が黙って相手の変更を消す。
**共有をやめる以外に解決手段がない。**

### 手順

`UserPromptSubmit` hook（`scripts/mark-prompt.js`）が、分離が必要なときだけ
次のような指示をコンテキストへ入れる。**指示があればそれに従う。**

```
EnterWorktree(name: "codex-<セッションのslug>")     ← 新規作成
EnterWorktree(path: "<既存worktreeの絶対パス>")     ← 再開・再利用
```

- **調査・質問だけで終わるなら移動しなくてよい。** worktreeは最初の書き込みで初めて要る
- 実装指示かどうか自信が持てないときは、**移動する側に倒す**。無駄なworktreeは残るだけだが、
  共有ツリーへの書き込みは他セッションの作業を消す
- 移動せずに共有作業ツリーへ書き込もうとすると、`PreToolUse` hook
  （`scripts/worktree-guard.js`）が**Edit / Write / NotebookEditを拒否する**。
  拒否メッセージに実行すべきEnterWorktreeが入っているので、それに従って移動し、
  同じ書き込みをやり直す
- **すでにlinked worktree内にいる場合は、さらにworktreeを作らない**（hookも何も言わない）
- 同じセッションでは常に同じworktreeになる（名前はsession_idから導出する）。
  別セッションは必ず別worktreeになる

### 共有作業ツリーに未コミット変更がある場合

**分離できないので、勝手に進めない。** worktreeはHEADから作られるため、ここで移動すると
その未コミット変更は共有ツリーへ取り残され、このセッションからは見えなくなる。

次は**絶対に行わない**（利用者の変更を壊すため）:

- `stash` / `reset` / `checkout`
- 未コミットファイルのコピー・移動
- 既存差分の削除、別セッションの変更の新worktreeへの移植

ユーザーへ次のどちらかを確認し、決まるまでファイルを変更しない。

1. 未コミット変更をコミットしてから、あらためて実装を指示する
2. このリポジトリでは分離しない（`.codex-review-no-worktree` を作成する）

運用が始まれば各セッションは編集前にworktreeへ入るので、共有ツリーはクリーンに保たれ、
この確認は基本的に初回だけになる。

### やらないこと

**push / rebase / worktreeの削除 / ブランチの削除は行わない。**
コミット・merge済みのworktreeも、利用者が確認するまで残す。
`ExitWorktree`も自分から呼ばない（ユーザーが明示的に求めたときだけ）。

**mergeはStep 7でだけ行う。** 自動コミットの直後に、`scripts/merge-reviewed.js`で
元のブランチへ1回merge する。それ以外の場面で自分から`git merge` / `git rebase` /
`git pull`を打たない（実装中に元ブランチを取り込みたくなっても、手で打たずに報告する）。

## 前提

- OpenAI Codex CLI が使えること（PATH 上の `codex`、または VS Code 拡張
  「Codex – OpenAI」同梱の `codex.exe`）。認証済みであること（`~/.codex/auth.json`）
- レビュー対象が git リポジトリであること
- レビューは読み取り専用サンドボックス（`-c sandbox_mode="read-only"`）で行う。
  **修正は必ず Claude 自身が行い、Codex には書き込ませない**
  （`--dangerously-*` 系フラグは絶対に付けない。フラグ省略時は workspace-write で
  動くため、read-only の明示を省略しない）

## 引数

| 呼び出し例 | 動作 |
| --- | --- |
| `/codex-review` | 未コミット差分（staged / unstaged / untracked）をレビュー |
| `/codex-review base <branch>` | 指定ブランチとの差分をレビュー |
| `/codex-review commit <sha>` | 指定コミットの変更をレビュー |
| `/codex-review setup` | モデル / エフォート設定のみ実施（レビューはしない） |
| `/codex-review config` | 現在の設定を表示（`scripts/config.js --show`） |
| `/codex-review effort <値>` | 明示指定したモデルのエフォートだけ変更 |
| `/codex-review auto` | **時刻でモードを自動判定する**ようにする（推奨・後述） |
| `/codex-review weekend` | 離席モードへ手動切り替え（後述） |
| `/codex-review weekday` | 通常モードへ手動で戻す（後述） |
| `/codex-review pending` | 離席中に持ち越した指摘と判断を表示（レビューはしない） |
| `/codex-review adopt-legacy` | 旧形式のrepository単位持ち越しを、確認済みの現在ブランチへ移行 |

**このスキルに実装作業は依頼できない。** レビュー専用であり、`codex exec review` は
スコープ指定とカスタム指示を併用できない（実機確認済み）。実装は普通に依頼し、
その差分をこのスキルがレビューする。

### `/codex-review auto`（推奨）

**端末の時刻から、通常モードと離席モードを毎回自動で判定する。**
切り替えコマンドを打つ必要がなくなる。

1. 自動レビューが未設定なら、対象リポジトリで次を実行する:

   ```
   node "<このスキルのdir>/scripts/setup-auto.js"
   ```

   `UserPromptSubmit` / `Stop` hook、`.gitignore`、フラグ、既定スケジュールを
   既存設定を保持して登録する。
   旧グローバル・project hookを検出して停止した場合は、他のリポジトリへの影響を説明し、
   ユーザーが移行を承認したときだけ`--migrate-legacy-hooks`を付けて再実行する。
2. 勤務日・勤務時間をユーザーに確認し、次で保存する:

   ```
   node "<このスキルのdir>/scripts/config.js" --preset auto
   node "<このスキルのdir>/scripts/config.js" --set schedule.workdays=1,2,3,4,5 --set schedule.start=09:00 --set schedule.end=18:00
   ```

既定は月〜金 9:00〜18:00。
設定後は `--show` の「今は」行で、現在どちらのモードかを見せる。

| | 勤務日 × 勤務時間内に**指示** = **通常モード** | それ以外に**指示** = **離席モード** |
| --- | --- | --- |
| 自動レビュー | 走る | 走る |
| 指摘の処理 | P0/P1/P2は必須修正、P3は対象外 | 同左（質問せず進める） |
| レビュー回数 | `rounds.work`（既定 3） | `rounds.away`（既定 2） |

- **時刻はレビューするかどうかを決めない。決めるのは「判断を質問できるか」と「何回まで」だけ。**
  `.codex-review-auto` があれば、時間帯に関係なく未レビューの差分はレビューされる
- **判定に使うのは「指示した時刻」であって、レビュー時点の時刻ではない。**
  実装に時間がかかって境界をまたいでも、指示した人がまだ席にいるなら聞く
- **判定は毎回その場で行い、設定には書き込まない。** 「週末に切り替えたまま月曜に
  戻し忘れて、無人のまま修正が入る」事故を構造的に無くすため
- スケジュールの指定が壊れている場合は、**安全側（通常モード＝必要な判断を人に聞ける）に倒れる**

### `/codex-review weekend` / `weekday`

スケジュールを使わず手で切り替える場合。**設定の書き換えは
`scripts/config.js` が行う**（値の検証と原子的な保存を含む。手で JSON を編集しない）。
どちらのプリセットも `schedule.enabled` を `false` にする（手動が自動を上書きする）。

`weekend`:

1. `node "<このスキルのdir>/scripts/config.js" --preset weekend`
   （無人モード ON）
2. 明示モデルを使っている場合だけ、**エフォートを下げるか確認する。**
   高いエフォートは 1 レビューが長く、
   レート制限を引き寄せる。`--show` が出す「選べる値」から選んでもらい、
   `--set effort=<値>` で保存する（**値をこの手順書に書かない**。モデルによって違う）
3. 設定内容を復唱して、離席してよい旨を伝える

`weekday`:

1. `node "<このスキルのdir>/scripts/config.js" --preset weekday`
2. **持ち越しファイルに項目が残っていれば、続けて通常レビューを提案する**
   （回数上限・修正失敗などで未解決のP0/P1/P2と、無人中の`decisions`を確認する）

どちらも `.codex-review-auto` には触らない。自動レビューを止めるかどうかは
モードとは独立した判断なので、ユーザーが明示的に求めたときだけ削除する。

### `/codex-review pending`

次を実行し、`deferred`と`decisions`を日本語で一覧表示する。空なら「持ち越しなし」と伝える。
この操作は読み取り専用で、ブランチID、`.git/config`、状態ファイルを作成・更新しない。

```
node "<このスキルのdir>/scripts/stop-hook.js" --pending
```

`legacy`がnullでなければ、旧形式のrepository単位持ち越しとして別枠表示する。
`legacy.error`がnullでなければ移行せず、壊れた旧状態としてエラーを報告する。
旧版が保存したP3は現行方針の対象外なので表示しない。
どのブランチ由来かを自動推測しない。元の作業ブランチを確認してから
`/codex-review adopt-legacy`を実行するよう案内する。

### `/codex-review adopt-legacy`

現在のブランチが旧形式の持ち越しを記録したブランチだとユーザーが確認済みの場合だけ、
次を実行する。新しいブランチ別ファイルへ重複除去して移し、成功後に旧ファイルを削除する。

```
node "<このスキルのdir>/scripts/stop-hook.js" --adopt-legacy-pending
```

ブランチを推測して自動実行しない。移行後は`/codex-review pending`で結果を表示する。

## 実行手順

### Step 0: 環境を確認する

```
node "<このスキルのdir>/scripts/doctor.js"
```

`READY:` で終われば続行する。`NOT_READY:` なら、出力に示された不足を解消してから進む。

- **`codex CLI` が無いだけなら `--fix` を付けて再実行してよい**（`npm i -g @openai/codex` を実行する）。
  コンテナは使い捨てなので問題ないが、**ホストではグローバルに入るのでユーザーに確認する**
- **認証は自動で解決できない。** 対話ログイン・認証情報の受け渡しのいずれかが要る。
  doctor の出す3つの選択肢を提示して、ユーザーに選んでもらう
- 設定JSON自体が壊れている場合は、ユーザー確認後に
  `node "<このスキルのdir>/scripts/config.js" --reset`を実行する。
  元ファイルは同じディレクトリへバックアップされる
- コンテナで動かす場合の詳細は `references/container.md` を読む

### Step 1: 設定を読む

`~/.claude/codex-review.config.json` を読む。形式:

```json
{
  "mode": "inherit",
  "rounds": { "work": 3, "away": 2 },
  "timeout_minutes": 15,
  "retry": { "max_attempts": 3, "wait_seconds": [60, 300, 900] },
  "unattended": { "enabled": false, "auto_fix": ["P0", "P1", "P2"] },
  "schedule": { "enabled": false, "workdays": [1,2,3,4,5], "start": "09:00", "end": "18:00" },
  "auto_commit": true
}
```

`unattended.auto_fix`と`auto_commit`は互換性のため残す固定値で、利用者が変更する
設定ではない。旧設定は読み込み時にP0/P1/P2・自動コミットONへ移行する。

**現在のモードと回数は設定値を直接読んで判断しない。**
Stop hookのblock指示に`--session-key`付きのコマンドがあれば、そのコマンドをそのまま実行する。
手動起動などで提示がない場合だけ次を実行する:

```
node "<このスキルのdir>/scripts/stop-hook.js" --mode
```

「通常モード / 離席モード」「重要度ごとの処理」「レビュー上限回数」が返るので、
それに従う。

通常モード・離席モードのどちらでも**レビューを始める前に**次も実行し、
`deferred` / `decisions` / `path`と`revision`を保持する。離席モードで新しい持ち越しを
保存するときにも`path`が必要であり、新規指摘が0件でも週末の持ち越しを
見落とさないため、後回しにしない。

```
node "<このスキルのdir>/scripts/stop-hook.js" --prepare-pending
```

`error`がnullでなければ状態を上書きせず、エラーをユーザーへ報告する。
`legacy`がnullでなければ旧形式のrepository単位持ち越しが残っている。
`legacy.error`がnullでなければ移行せずエラーを報告する。
離席モードでは自動移行せず完了報告へ残す。通常モードでは現在のブランチ由来かを確認し、
確認できた場合だけ`--adopt-legacy-pending`を実行してから`--prepare-pending`をやり直す。
通常モードでは、ここで読んだ`deferred`のコピーを`未解決持ち越しリスト`として初期化する。
以降のラウンドでは元のStep 1スナップショットを再追加せず、このリストだけを更新する。

**判定は「ユーザーが指示した時刻」で行われる**（`scripts/mark-prompt.js` が
UserPromptSubmit hook で記録している）。レビュー実行時の時刻ではない。
17:50 に指示して実装が 18:10 に終わっても、まだ席にいるので必要な判断を質問できる、
という挙動のため。記録が無い場合のみ現在時刻にフォールバックする。

- `mode: "explicit"` … `-m <model>`を付けて実行する
- `mode: "inherit"` … `-m`を付けず、Codex 本体の`~/.codex/config.toml`からモデルを継承する
- `effort` … `explicit`で設定されている場合だけ`-c model_reasoning_effort="<effort>"`を付ける。
  `inherit`ではモデルとeffortの両方をCodex本体から継承する
- `retry` … レート制限（429）で落ちたときの再実行。`scripts/run-review.js` が処理する
- `unattended` … 無人実行モード（後述）
- `schedule` … 時刻でモードを自動判定する設定（後述）
- `auto_commit` … 互換用の固定値`true`。このスキルの完了時だけ自動コミットする
- ファイルが存在しない場合も既定値（`mode: "inherit"`）で続行する。
  休日の初回タスクをモデル選択待ちで止めない

**モデル名やエフォートを SKILL.md や会話にハードコードしないこと。**
必ずこの設定ファイルとモデルキャッシュ（後述）から取得する。

**設定の書き換えは `scripts/config.js` で行う。** JSON を直接編集しない
（値の検証と原子的な保存をスクリプトが担当する）。

codex CLI の解決（PATH → VS Code 拡張同梱）、モデル/エフォートのフラグ組み立て、
サンドボックス指定、レート制限の再実行はすべて `scripts/run-review.js` が行う。
**Claude が `codex` を直接呼ばないこと。**

### Step 2: レビューを実行する

1. 対象を確認する:
   - 既定（`--uncommitted`）: `git status --short` で未コミット差分を確認する。
     差分がなければその旨を伝えて終了
   - `base <branch>`: 次を**両方**確認する。
     1. `git rev-parse --verify <branch>` — ブランチが存在すること
     2. `git status --porcelain` が空（作業ツリーがクリーン）であること —
        未コミット変更が残っていると、1 周目はブランチ間差分だけ・2 周目以降と
       レビュー済み記録は作業ツリー全体が対象になり、依頼外の変更を修正候補にしたり、
        未レビューの内容をレビュー済みと記録したりする
     → クリーンでないときは **修正まで進めず、レビュー結果の報告だけを行う**
       （このとき Step 7 の確定処理も実行しない）
   - `commit <sha>`: 次の 3 つを**すべて**確認する。
     1. `git rev-parse --verify "<sha>^{commit}"` — そのコミットが存在すること
     2. `git rev-parse HEAD` と一致すること — Step 5 の修正は「そのコミット」ではなく
        現在チェックアウト中の作業ツリーに当たるため、別リビジョンだと無関係なコードを壊す
     3. `git status --porcelain` が空（作業ツリーがクリーン）であること —
        未コミット変更が残っていると、1 周目はコミットだけ・2 周目以降とレビュー済み記録は
        作業ツリー全体が対象になり、依頼外の変更を修正候補にしたり、
        未レビューの内容をレビュー済みと記録したりする
     → いずれかを満たさないときは **修正まで進めず、レビュー結果の報告だけを行う**
       （このとき Step 7 の確定処理も実行しない）。修正が必要なら、対象コミットの
       チェックアウト / 作業ツリーをクリーンにする / `git worktree` の用意 の
       いずれかをユーザーに依頼する（勝手にチェックアウトや stash をしない）
2. **レビュー対象の fingerprint を控える**（Step 7 で使う。ラウンドごとに取り直す）:

   ```
   node "<このスキルのdir>/scripts/stop-hook.js" --print
   ```

   レビューは数分かかる。その間に作業ツリーが変わると（自分の追加編集、
   **同じツリーを共有する別セッション**）、Codex が読んでいない変更まで
   「レビュー済み」と記録してしまう。それを防ぐため、レビュー開始直前の状態を控える
3. **レビュー実行中は作業ツリーを触らない。** 待っている間に別の作業を進めない
4. 出力先はセッションの scratchpad ディレクトリにする（リポジトリ内に置かない）
5. カレントディレクトリを対象リポジトリのルートにして実行する:

```
node "<このスキルのdir>/scripts/run-review.js" --uncommitted --out "<scratchpad>/codex-review-round<N>.txt"
```

- 対象が `base` / `commit` 指定なら `--uncommitted` を `--base <branch>` /
  `--commit <sha>` に置き換える
- **必ずバックグラウンドで実行する。** レート制限に当たると設定の `wait_seconds` ぶん
  待ってから再実行するので、1 回の呼び出しが数十分に及びうる。
  呼び出し側のタイムアウトは
  `timeout_minutes × retry.max_attempts + sum(retry.wait_seconds)` 以上にする
- stdout の最終行が結果（`OK: ...` / `NG: ...`）。**指摘は `--out` のファイル**から読む
- `NG:` で終わったら**ループを中断**し、理由をユーザーに報告する
  （Step 7の確定処理もしない）
- **重要（実機確認済みの制約）**:
  - `--uncommitted` / `--base` / `--commit` はカスタムプロンプト `[PROMPT]` と併用できない
    （「cannot be used with '[PROMPT]'」エラーになる）。レビュー観点やP3除外方針を
    Codex に指示することはできない
  - `--output-schema` は review モードでは無視され、結果は Codex 標準のテキスト形式で返る。
    構造化は Step 3 で Claude が行う
- P3の除外は Step 3 の Claude 側フィルタで行う
  （Codex が同じ指摘を再報告してくることは織り込み済み）

### Step 3: 結果をパースして判定する

`-o` のファイル（テキスト）を読む。おおむね次の形式で返る:

```
<総評（1〜数行）>

Review comment:

- [P1] <タイトル> — <ファイルパス>:<開始行>-<終了行>
  <本文（何が問題か・どう直すべきか）>
```

- `[P0]`〜`[P3]`の優先度・タイトル・ファイル・行・本文を抽出し、指摘ごとに連番idを
  振って内部リストにする（形式が多少揺れても趣旨で読み取る）。重要度は影響で判断し、
  次を基準にする:
  - `P0`: アプリが起動しない、データが消える、セキュリティホールがある
  - `P1`: 特定条件でクラッシュする、APIが正常に動かない、大きなメモリリークがある
  - `P2`: エラーハンドリング不足、パフォーマンスを改善できる、重複コードがある
  - `P3`: コメント改善、分かりやすい命名、フォーマット、任意のリファクタリング提案
- **P0を取りこぼさないこと。P0/P1/P2は必須修正、P3は対象外。** P3は完了報告へ
  一覧表示するが、修正せず、`deferred`にも保存しない
- 出力は英語のことが多い。**報告では Claude が日本語に訳して提示する**
- 旧版の見送りリストにP0/P1/P2があっても除外しない。現行方針では必須修正である。
  P3は見送りリストとの一致に関係なく対象外にする
- 通常モードでは、現在の`未解決持ち越しリスト`からP3を除き、残ったP0/P1/P2を
  新規指摘へ統合する。
  同趣旨なら1件へまとめる。**Step 1で読んだ元の`deferred`をラウンドごとに
  再追加しない**
- P0/P1/P2の統合後の指摘が0件ならStep 7へ、1件以上ならStep 4へ進む。
  Codexの新規指摘が0件でも`未解決持ち越しリスト`が空でなければStep 4へ進む

### Step 4: 対応する指摘を決める

1. P0/P1/P2を優先度順にすべて**対応リスト**へ入れる。採否確認は行わない
2. P3を**対象外リスト**へ入れる。修正・再レビュー理由・持ち越しには使わない
3. 通常モードで、必須修正の方法に要件を変える複数案があり、既存設計から決められない
   場合だけAskUserQuestionで方針を確認する。指摘自体を見送る選択肢は出さない
4. 離席モードではAskUserQuestionを使わず、後述の判断規則に従う

持ち越し由来のP0/P1/P2は、Step 5で修正に成功するまで
`未解決持ち越しリスト`に残す。

### Step 5: 対応リストを修正する

**このラウンドが上限（現在のモードの `rounds`）の回目なら、修正せず、対応リストを
現在モードの未解決リストへ加えて Step 7 へ進む。**
ここで直すと作業ツリーが変わり、その変更を Codex は一度も読んでいないため、
Step 7 の確定処理が拒否されて「もう一周」になる。つまり上限が上限として機能しない。
必須修正のP0/P1/P2は、未修正である限り`deferred`へ保存する。P3は保存しない。

- 離席モード: `離席未解決リスト`へ加える。ラウンド途中では保存せず、
  最終レビュー後にStep 7で1回だけ追記する
- 通常モード: `未解決持ち越しリスト`へ加え、Step 7で1回だけ
  置換する

持ち越しの保存とレビュー済み記録はStep 7の`--finalize-pending`が同じfingerprintに
結びつけて行う。失敗した場合は未解決の指摘を隠すより、レビュー済みと偽らないことを
優先する。作業ツリーが変わっていなければ、Step 2で控えたfingerprintを使って次を実行し、
次のユーザー指示時に再レビューできるよう予約する:

```
node "<このスキルのdir>/scripts/stop-hook.js" --retry <控えた fingerprint>
```

`--retry`にも失敗した場合は、保存エラーと手動復旧が必要なことを完了報告へ残す。

確定できた指摘は **🔧 要対応（未レビュー）として報告に回す**。

- 対応リストの指摘だけを、最小の差分で修正する
- 指摘に便乗した無関係なリファクタリングをしない
- suggestion が既存設計と矛盾する場合は、設計に合う形に読み替えて修正し、その旨を報告に含める
- 通常モードで持ち越し由来の項目を実際に修正できたら、
  `未解決持ち越しリスト`から除く。次のCodexレビューで同趣旨が再報告された場合は、
  修正不十分の新規指摘として再び未解決にする

### Step 6: 再レビュー（ループ）

Step 2 に戻る。ラウンド数をカウントし:

- **2 ラウンド目以降のスコープ**: Claude の修正は未コミットの作業ツリーに入るため、
  初回が `base` / `commit` 指定だった場合も **2 回目以降は `--uncommitted` に切り替えて**
  修正を含む状態をレビューさせる（切り替えた旨を完了報告に含める）。
  切り替えないと Codex は元のスナップショットを見続け、修正が一切検証されない
- 対応済みのP0/P1/P2が再び報告された場合は「修正が不十分」とみなし、
  対応リストと`未解決持ち越しリスト`へ戻す
- 新規のP0/P1/P2が0件（P3だけの場合を含む）→ Step 7
- 上限（現在のモードの `rounds`）に達したら打ち切り、未解決の指摘を持ち越しへ
  保存して報告に含める

### Step 7: レビュー済みとして記録し、完了報告する

1. **最後のレビューが `--uncommitted` だったなら必ず**、持ち越し状態とレビュー済み記録を
   1回の`--finalize-pending`で確定する。Step 2で控えたfingerprintは
   **最後のラウンドのもの**を使う。

   通常モードは現在の`未解決持ち越しリスト`、Step 1の`decisions`、`path` / `revision`を
   含む次のJSONをscratchpadへ作る。リストが空でも、修正済み・対象外になった古い項目を
   消すため省略しない:

   ```json
   {
     "strategy": "replace",
     "fingerprint": "<最後のラウンドで控えたfingerprint>",
     "path": "<--prepare-pendingで得た値>",
     "revision": "<--prepare-pendingで得た値>",
     "deferred": [],
     "decisions": []
   }
   ```

   離席モードは最終ラウンドの`離席未解決リスト`と、全ラウンドで積み上げた
   `離席判断リスト`を含む次のJSONを作る:

   ```json
   {
     "strategy": "append",
     "fingerprint": "<最後のラウンドで控えたfingerprint>",
     "path": "<--prepare-pendingで得た値>",
     "deferred": [],
     "decisions": []
   }
   ```

   次を1回だけ実行する:

   ```
   node "<このスキルのdir>/scripts/stop-hook.js" --finalize-pending "<一時JSON>"
   ```

   - スクリプトはfingerprintを持ち越し更新の前後とレビュー済み記録の直前に確認する。
     作業ツリー、ブランチ、pending revisionのどれかが変わっていれば、持ち越しを
     元へ戻してレビュー済み記録もしない。そのときは**現在の差分を対象にStep 2からやり直す**
   - 保存や状態更新そのものに失敗し、作業ツリーがまだ同じ場合はfingerprintを
     `--retry`へ渡す。古い持ち越しを残したままレビュー済みにしない
   - report-onlyで終わった`commit` / `base`モードでは実行しない
   - 指摘ゼロ、上限打ち切り、P3だけの場合でも必ず実行する

2. **`--finalize-pending`が成功したら、後述の「コミットの出し方」に従って
   レビュー対象を自動コミットする。** コミット結果を控える。

3. **コミットの成否にかかわらず、後述の「mergeの出し方」に従って元のブランチへmergeする。**
   merge結果を控えてから報告へ進む。

4. **次の形式で締める。この形式はこのスキル専用。**

   利用者は複数セッションを並べて表示しており、**どれが終わったかを一目で判別する**
   のがこの形式の目的。**見出しと絵文字は毎回まったく同じ文字列を使う。**
   言い換えない、気を利かせて変えない、新しい絵文字を発明しない。
   ブレた時点でこの形式の意味が無くなる。

   ```
   ## ✅ codex-review 完了

   <何件指摘され、何を直したか。1〜2 文>

   ### 🌳 ブランチ

   - worktree : <worktreeの絶対パス>
   - ブランチ : <ブランチ名>
   - 基準     : <基準コミットのhash>
   - 統合     : <merge先ブランチとmerge-reviewed.jsの結果。mergeしていない場合はその理由>（pushはしていません）

   ### 残タスク

   - ⚠️ 未検証 — 確認できていないこと と その理由
   - 🔧 要対応 — 未解決の指摘（回数上限での打ち切りなど）
   - 💤 対象外 — 修正しないP3の指摘（id / title / 重要度）
   - 🚫 保留 — 無人モードで実行しなかったもの（破壊的操作など）
   - ❓ 課題 — 無人モードで自分が判断した箇所（何を・なぜ・選ばなかった案）

   ### 📦 コミット

   <通常・離席共通: コミットハッシュ、変更なし、または安全上コミットできなかった理由>
   ```

   - **mergeできなかった場合は、`- 🔧 要対応` にその理由と、利用者がやるべきことを出す。**
     コミットしただけでは共有チェックアウトに反映されず、サイトや別セッションからは見えない

   - **`## ✅ codex-review 完了` は完全固定。** この行だけは一字も変えない
   - 残タスクは**該当する区分の行だけ**残す。ひとつも無ければ `- なし` とだけ書く
   - 絵文字は上記の 8 種（✅🌳⚠️🔧💤🚫❓📦）だけ。**それ以外は使わない**
   - **対応しなかった指摘は黙って落とさない。** P3の対象外・未解決持ち越しとも必ず一覧に出す
   - 本文には往復回数・使用モデル・エフォート・レート制限での再試行回数も添える
   - **この形式を使うのはレビューのループを終えたときだけ。** 途中経過には使わない

   `### 🌳 ブランチ` の値は次で取得する。**推測で書かない。**

   ```
   git rev-parse --show-toplevel        # worktreeの絶対パス
   git rev-parse --abbrev-ref HEAD      # ブランチ名
   git merge-base HEAD <元のブランチ>    # 基準コミット（元ブランチが分かる場合）
   ```

   - worktree分離をしていない場合（読み取り専用・`.codex-review-no-worktree`・
     分離できず共有ツリーのまま進めた場合）は、`### 🌳 ブランチ` に
     **「worktree分離なし」とその理由**を書く。分離したことにしない
   - `EnterWorktree`が使えない環境で分離できなかった場合も同様に明記する
   - **merge結果を必ず残す。** コミットしただけでは共有チェックアウトへ反映されない
   - **pushしていないことを必ず残す。** リモートへの反映は人が確認して行う

### 📦 コミットの出し方

**`### 📦 コミット` は省略しない。**

- 通常・離席モードとも、Step 7の`--finalize-pending`成功後にレビュー対象を自動コミットする
- このスキルを起動したことをコミットの明示依頼として扱う。スキル外の作業はコミットしない
- **pushは絶対にしない**

最終レビューのfingerprint、コミットメッセージ、対象ファイルを次へ渡す。
対象は`git status --short --untracked-files=all`で確認し、ディレクトリではなく個別ファイルを列挙する。

```
node "<このスキルのdir>/scripts/commit-reviewed.js" --expected <最後のfingerprint> --message "<メッセージ>" -- <対象ファイル...>
```

- **メッセージはそのリポジトリの既存の慣習に合わせる。** `git log --oneline -10` を見て、
  gitmoji の有無・言語・粒度を揃える。**このスキルの好みを持ち込まない**
- 対象は**今回のレビュー範囲のファイル**。無関係な変更が混ざっているなら、
  対象へ渡さず、作業ツリーに残ったことを報告する
- コミットすべきでない変更（生成物・一時ファイル・秘密情報の疑い）があれば、
  コミットせず理由を報告する
- 対象リポジトリの上位指示がAIのコミットを明示的に禁止している場合は従い、実行しない
- `COMMITTED <hash>`なら成功、`NO_CHANGES`ならコミット対象なしとして報告する
- スクリプトは最終レビュー後の差分変更、対象外のステージ済みファイル、対象ファイルの
  ステージ中変更を検出したらコミットしない。**勝手にresetやunstageをしない**
- コミットに失敗し、現在も差分がある場合は、現在のfingerprintを`--print`で取り直して
  `--retry`へ渡す。レビュー済み扱いで未コミット差分を埋もれさせない

### 🔀 mergeの出し方

**worktree内のコミットは、そのworktreeの外からは見えない。** 共有チェックアウトは自分の
HEADを持ち続けるため、そこから配信しているサイトや、そのブランチを見ている利用者には
**専用ブランチへコミットしただけでは何も届かない**。コミットは半分で、mergeが残り半分。

自動コミットの直後に、worktree内（＝現在のcwd）から次を1回実行する。引数はない。

```
node "<このスキルのdir>/scripts/merge-reviewed.js"
```

merge先は**worktreeを作った時点の元のブランチ**。`UserPromptSubmit` hook
（`scripts/mark-prompt.js`）が、分離を指示する時点の共有チェックアウトのブランチを
`git config --local codexreview.<worktree名>.mergeInto`へ記録し、merge時はその記録と
共有チェックアウトの現在ブランチが**一致するときだけ**mergeする。

**merge先を現在のブランチから推測しない。** セッションが動いている間に利用者が
`git switch`することはあり、そのとき「今checkoutしているブランチ」は
「この作業がどこから来たか」の答えではない。推測すると、レビュー済みの変更が
無関係なブランチへ入る。

一致を確認したうえで、スクリプトが共有チェックアウト側で
`git merge --ff --no-edit -m "Merge branch '<ブランチ>'" refs/heads/<このセッションのブランチ>`
を実行する。fast-forwardできるならmergeコミットは作らない（`merge.ff = only`のリポジトリでも
同じ挙動になるよう`--ff`を明示している）。

ブランチ名は常に`refs/heads/`付きで渡す。どんな名前でもgitのオプションとして解釈されず、
同名タグに横取りされることもない（gitの解決順ではタグがブランチより優先される）。
そのぶんmergeコミットの件名が`Merge branch 'refs/heads/…'`になってしまうため、
件名はgitの慣習どおりの文面を明示的に渡している。

**共有チェックアウトのindexと作業ツリーはリポジトリ全体で1つしかない。** そのため
前提の再確認からmerge・必要なabort・最終確認までを、**リポジトリ共通gitディレクトリ単位の
ロック**の下で実行する（worktree単位のロックでは、別worktreeのセッションを排他できない）。
ロックを取れなければ、共有チェックアウトを変更する操作を一切始めずに失敗する
（ロック取得前に行うのは読み取りだけ）。
また、**自分が開始していないmergeは絶対にabortしない**（`MERGE_HEAD`が自分のsourceと
一致する場合だけabortする）。

**merge後にコミットしない。** 自動コミットは既に終わっており、fast-forwardなら
レビュー済みのコミットがそのままmerge先の先端になり、分岐していればgitがmergeコミットを作る。
merge後に残っている変更はレビュー対象外なので、**巻き込んでコミットしない**。

- `MERGED <ブランチ> <hash> <fast-forward|merge-commit>` … 反映済みとして報告する
- `UP_TO_DATE <ブランチ>` … merge済み・またはコミットが無い。そのまま報告する
- `SKIPPED not-linked-worktree` … worktree分離していない（コミット時点で現在のブランチに入っている）
- `SKIPPED same-branch` … worktreeが統合先ブランチ自体をcheckoutしている（移すものがない）
- 失敗（終了コード 1）… **理由をそのまま `- 🔧 要対応` に出す。** 握りつぶさない

次の場合、スクリプトはmergeせずに失敗する。**自分で回避しようとしない**
（`stash` / `reset` / `checkout` / `-X ours` / `--strategy` での強行はすべて禁止）。
どれも「共有チェックアウトを元のまま返す」ことを優先した結果であり、失敗しても
worktree内のコミットは残っているので、利用者が対処すれば統合できる。

- **共有チェックアウトが、記録された統合先と別のブランチをcheckoutしている**
  → 統合先をcheckoutし直すよう利用者へ依頼する。**別のブランチへmergeし直さない**
- 統合先が記録されていない / 複数記録されている / ブランチ名として不正
  → 記録前に作られたworktreeか、記録が壊れている。エラーに復旧コマンドが出る
- 記録された統合先ブランチが存在しない（削除・改名された）
- 共有チェックアウトに未コミットの追跡ファイル変更がある → コミットか退避を利用者へ依頼する
- 共有チェックアウトがdetached HEAD、またはmerge / rebase途中
  （**他プロセスのmergeはabortしない**。そのまま報告する）
- mergeが衝突する → スクリプトが`--abort`して元へ戻し、HEAD・index・作業ツリーが
  実行前と一致することを確認する。**衝突の解決は利用者に任せる**
- 他セッションのmergeロックを60秒待っても取れない

- **pushは絶対にしない。** merge後もリモートへは出さない
- 対象リポジトリの上位指示がAIのコミット・mergeを明示的に禁止している場合は従い、実行しない
- 離席モードでもmergeする。判断を仰ぐために止めない（失敗時は報告に残るだけで、
  共有チェックアウトは元のまま変わらない）

### 自動コミット（`auto_commit`、固定 `true`）

`auto_commit`は旧設定との互換性のためだけに残す。通常・離席モードとも上のスクリプトを
実行し、利用者設定でOFFにはしない。

- **`push` は絶対にしない。** 人が確認して実行する
- **未解決の指摘が残ったままコミットする場合は、その旨を報告に明記する**
  （🔧 要対応 / 💤 対象外 に出す）。コミットしたことが「全部直った」を意味しない

## モデルの再設定（`/codex-review setup`）

**ここでも選択肢は 1 質問につき 2〜4 個の制約がかかる。**

1. Codex homeの`models_cache.json`を読む（`CODEX_HOME`設定時はその配下、
   未設定なら`~/.codex/models_cache.json`）。`models`から`visibility == "list"`のものを
   `priority` 順に取り出す（slug / display_name / supported_reasoning_levels）
2. AskUserQuestion でモデルを選択させる:
   - 選択肢は上位 3 モデル ＋「Codex 本体の設定を継承」（= `mode: "inherit"`）の計 4 個
   - キャッシュが読める場合、一覧にないモデルslugは選ばない
3. `explicit` の場合、選択したモデルの `supported_reasoning_levels` から
   エフォートを選択させる（そのモデルが対応する値だけを選択肢に出す）。
   **エフォートが 5 段階以上あるモデルが実在する**（`low` / `medium` / `high` /
   `xhigh` / `max` / `ultra` の 6 段階など）。その場合は全部を並べず、
   **代表的な 3 個に絞り、残りは Other で入力してもらう**
4. **保存は必ずスクリプトで行う**（JSON を直接書かない）:

   - 継承を選んだ場合:

     ```
     node "<このスキルのdir>/scripts/config.js" --set mode=inherit --set effort=
     ```

   - モデルを選んだ場合:

     ```
     node "<このスキルのdir>/scripts/config.js" --set mode=explicit --set model=<slug> --set effort=<値>
     ```

   値は保存前に検証される。保存後の表示を復唱する
5. `models_cache.json` が存在しない・読めない場合は、モデル slug と
   エフォートを自由入力してもらう（検証はスキップされ、次回実行時のエラーで判明する）

## モデルが廃止・変更されたとき

- `codex exec` がモデル不明・非対応エラーを返したら、ループを中断する
- Codex homeの`models_cache.json`を読み直して現行モデルを提示し、セットアップを再実行する
  （`~/.codex/config.toml` の `[notice.model_migrations]` に後継モデルの対応表が
  ある場合は、それを推奨として提示する）
- 設定を更新したらレビューを再開する

## 離席モード（`--mode`が離席モード）

土日など人がいない間もレビューを回すためのモード。**ダイアログを出さず、重要度で
自動的に仕分ける。** `unattended.enabled`を手動で有効にした場合、または
`schedule`が指示時刻を勤務外と判定した場合だけこの経路に入る。

### 大原則: 無人時は判断を仰いで止まらない

**無人モードでは AskUserQuestion を一切使わない。** 人がいないので、聞いた時点で
週末がまるごと止まる。判断が必要になったら次のようにする。

1. **依頼の要件・既存の設計・このスキルの方針に照らして、最も近い選択肢を選んで進める**
2. 何をどう判断したか、**選ばなかった選択肢**とあわせて記録する（後述のファイル）
3. 完了報告で **「課題」として明示的に報告する**。勝手に決めた事実を埋もれさせない

これは修正方針の選択に限らず、無人中に発生するすべての判断に適用する。

**ただし次は無人でも実行せず、保留して記録・報告に回す**（要件に近いかどうかに関わらず）:

- 破壊的な操作、データやファイルの削除、後戻りできない変更
- 依頼の要件を明らかに超える変更、無関係なリファクタリング
- プッシュ / 公開 / 外部への送信 / 課金を伴う操作
- ユーザーの既存の変更を捨てる操作

これらは「要件に近い判断」で押し切れる範囲を超えている。止まらないことより、
週明けに取り返しがつくことを優先する。

### Step 4 の代わりに行うこと

1. 最初のラウンドで、メモリ上に`離席未解決リスト`と`離席判断リスト`を空で作る。
   状態ファイルへはまだ書かない
2. 各ラウンドの指摘を重要度で二分する:
   - `P0` / `P1` / `P2`（固定）→ **対応リスト**
   - `P3` → **対象外リスト**。修正も持ち越しもしない
3. `離席未解決リスト`は、修正失敗・安全上の保留・上限打ち切りで残った
   P0/P1/P2だけにする。P3は入れない
4. 自動判断は`離席判断リスト`へ積み上げる。同じtopic / choseの判断は1件へまとめる
5. **対応リストが空なら、そこでループを終える**（Step 6 に戻らず Step 7 へ）。
   自動修正の対象が無いということは作業ツリーが変わらないので、再レビューしても
   同じ指摘が返るだけ。上限まで回すと、**無駄な API 呼び出しで
   レート制限を引き寄せる**（無人で放置する土日ほど効いてくる）
6. 上限ラウンドなら対応リストも未修正のまま`離席未解決リスト`へ加え、Step 7へ進む
7. 上限前なら対応リストをStep 5の手順で修正し、Step 6でループする。
   修正失敗・安全上の保留でループを続けられない項目は`離席未解決リスト`へ加える
8. 状態ファイルへの追記はラウンド途中に行わない。ループ終了後、Step 7の
   `--finalize-pending`で最終リストだけを1回追記する

### 有人に戻ったとき

Step 1の`--mode`が通常モードを返した実行では、Step 1で読んだ持ち越しを処理する。

- `deferred` のP0/P1/P2を、今回のCodexの指摘と合わせて必須修正する。
  旧版が保存したP3は対象外として除く。
  Step 1のコピーを`未解決持ち越しリスト`として持ち、ラウンド間はこのリストを更新する。
  元の`deferred`全件を毎回マージし直さない。
  **対応リストへ入れただけで全件を`deferred`から除かない。**
  - Step 5で実際に修正できた項目は除く
  - 最終ラウンド・修正失敗・安全上の保留で未修正なら残す
  - 今回の新規P0/P1/P2が未修正なら、この置換JSONの
    `deferred`へ直接追加して次回も提示できるようにする

  Step 5の結果が確定してから、残す項目と控えた`path` / `revision`を保持する。
  Step 7で`strategy: "replace"`と最後のfingerprintを加え、レビュー済み記録と
  一緒に確定する:

  ```json
  {
    "path": "<--prepare-pendingで得た値>",
    "revision": "<--prepare-pendingで得た値>",
    "deferred": [
      { "severity": "P2", "title": "<未解決項目>", "file": "...", "line": 0,
        "description": "...", "suggestion": "...", "recorded_at": "<既存値>" }
    ],
    "decisions": [
      {
        "topic": "既存の判断ログ",
        "chose": "選んだ内容",
        "because": "判断理由",
        "alternatives": ["選ばなかった案"],
        "recorded_at": "<既存値>"
      }
    ]
  }
  ```

  **`decisions`はStep 1で読んだ未確認項目をそのまま全件コピーする。**
  ユーザーが確認済みと明示するまで空配列にしない。

  状態ファイルを直接編集せず、ラウンド途中で`--replace-pending`もしない。
  ブランチが変わって`path`が一致しない場合、別セッションが先に追記して`revision`が
  変わった場合、作業ツリーがfingerprintと違う場合はStep 7の確定が拒否されるので、
  対象ブランチで`--prepare-pending`とレビューからやり直す
- `decisions` は**ダイアログではなく報告で見せる**（済んだ判断であって、選択肢ではない）。
  ユーザーが確認したと言うまで消さない。確認済みと明示された場合だけ、もう一度
  `--prepare-pending`で最新path / revisionを取得し、`deferred`を保持したまま
  `decisions: []`を含むJSONを`--replace-pending`へ渡して置換する。
  これはレビュー完了処理ではなく、確認済み判断ログだけを整理する操作
- 「今回の Codex の指摘」と「持ち越し」が同趣旨なら 1 つにまとめる

### 注意

- **無人モードでも `--dangerously-*` 系フラグは使わない。** Codex は読むだけ
- 無人で自動修正する以上、**誤検知や方針違いの指摘を取り込むリスクがある**。
  週明けに各セッションのコミットと完了報告を確認すること
- レート制限に当たったら `run-review.js` が待って再実行する。
  それでも `NG:` で終わったらループを中断し、状況を記録して報告に回す
  （待ち続けたり、叩き直したりしない）
- 手動の`weekend`プリセットを使った場合だけ、戻ったときに`weekday`へ切り替える。
  `auto`では時刻から毎回判定するため切り替え不要

## 自動レビューループ

`/codex-review` を手で打つ代わりに、Stop hook で「Claude が応答を終えようとしたら
未レビューの差分を検知して自動でレビューを回す」運用もできる。
セットアップは`scripts/setup-auto.js`が決定論的に行う。利用者が仕組み・複数セッション・
手動設定を知りたいときだけ`references/auto-loop.md`を読む。

有効なリポジトリでは3つのhookが動く。

| hook | スクリプト | 役割 |
| --- | --- | --- |
| `UserPromptSubmit` | `mark-prompt.js` | 指示時刻を記録し、必要ならworktree分離を指示する |
| `PreToolUse` | `worktree-guard.js` | 共有作業ツリーへのEdit / Write / NotebookEditを拒否する |
| `Stop` | `stop-hook.js` | 未レビュー差分を検知してレビューを指示する |

- 有効なリポジトリでは、ルートに `.codex-review-auto` が置かれている
- hook は判定するだけで、レビュー・重要度判定・修正はこのスキルが担当する
- そのため **Step 7 の `--finalize-pending`を飛ばすとhookが発火し続ける**
- worktree内でも同じ3つのhookが動く（`.worktreeinclude`がフラグと
  ローカル設定を複製する）。レビュー状態はworktreeごとに分かれる

## 注意事項

- Codex には書き込み・コマンド実行の権限緩和フラグを渡さない（レビューは読み取りのみ）
- レビュー結果ファイルや一時ファイルをリポジトリ内に作らない（scratchpad を使う）
- 認証エラー（未ログイン）の場合は、Codex 拡張または `codex login` でのログインを案内する
- 通常・離席モードともレビュー確定後に対象だけコミットする。pushはしない
