# codex-review — Codex 独立レビュー & 修正ループ

Claude が書いたコードを、別の AI（OpenAI Codex）に非対話モードでレビューさせ、
指摘の取捨選択はユーザーに委ね、採用された指摘だけを修正して再レビューする。

```
対象を確認 → codex exec review 実行 → 結果テキストを Claude が構造化
  → 指摘 0 件 …… 完了報告して終了
  → 指摘あり …… AskUserQuestion で指摘ごとに「対応 / 見送り」を選択
       → 対応分を修正 → 再レビュー（見送り分は Claude 側で除外）→ 先頭に戻る
  → 最大往復回数に達したら打ち切って報告
```

## 前提

- OpenAI Codex CLI が使えること（PATH 上の `codex`、または VS Code 拡張
  「Codex – OpenAI」同梱の `codex.exe`）。認証済みであること（`~/.codex/auth.json`）
- レビュー対象が git リポジトリであること
- レビューは読み取り専用サンドボックス（`-c sandbox_mode="read-only"`）で行う。
  **修正は必ず Claude 自身が行い、Codex には書き込ませない**
  （`--dangerously-*` 系フラグは絶対に付けない。フラグ省略時は workspace-write で
  動くため、read-only の明示を省略しない）

## 引数

| 呼び出し例 | 動作 |
| --- | --- |
| `/codex-review` | 未コミット差分（staged / unstaged / untracked）をレビュー |
| `/codex-review base <branch>` | 指定ブランチとの差分をレビュー |
| `/codex-review commit <sha>` | 指定コミットの変更をレビュー |
| `/codex-review setup` | モデル / エフォート設定のみ実施（レビューはしない） |

## 実行手順

### Step 0: 環境を確認する

```
node "<このスキルのdir>/scripts/doctor.js"
```

`READY:` で終われば続行する。`NOT_READY:` なら、出力に示された不足を解消してから進む。

- **`codex CLI` が無いだけなら `--fix` を付けて再実行してよい**（`npm i -g @openai/codex` を実行する）。
  コンテナは使い捨てなので問題ないが、**ホストではグローバルに入るのでユーザーに確認する**
- **認証は自動で解決できない。** 対話ログイン・認証情報の受け渡しのいずれかが要る。
  doctor の出す3つの選択肢を提示して、ユーザーに選んでもらう
- コンテナで動かす場合の詳細は `references/container.md` を読む

### Step 1: 設定を読む

`~/.claude/codex-review.config.json` を読む。形式:

```json
{
  "mode": "explicit",
  "model": "<モデルslug>",
  "effort": "<low|medium|high|xhigh など>",
  "max_rounds": 5,
  "timeout_minutes": 15,
  "retry": { "max_attempts": 3, "wait_seconds": [60, 300, 900] },
  "unattended": { "enabled": false, "auto_fix": ["P0", "P1"] },
  "auto_commit": false
}
```

- `mode: "explicit"` … `-m <model> -c model_reasoning_effort="<effort>"` を付けて実行する
- `mode: "inherit"` … モデル系フラグを付けず、Codex 本体の `~/.codex/config.toml` の設定を継承する
- `retry` … レート制限（429）で落ちたときの再実行。`scripts/run-review.js` が処理する
- `unattended` … 無人実行モード（後述）。既定は無効
- `auto_commit` … 完了時に自動でコミットするか（後述）。既定は無効。`push` は対象外
- ファイルが存在しない場合は「初回セットアップ」（後述）を実施してから続行する

**モデル名やエフォートを SKILL.md や会話にハードコードしないこと。**
必ずこの設定ファイルとモデルキャッシュ（後述）から取得する。

codex CLI の解決（PATH → VS Code 拡張同梱）、モデル/エフォートのフラグ組み立て、
サンドボックス指定、レート制限の再実行はすべて `scripts/run-review.js` が行う。
**Claude が `codex` を直接呼ばないこと。**

### Step 2: レビューを実行する

1. 対象を確認する:
   - 既定（`--uncommitted`）: `git status --short` で未コミット差分を確認する。
     差分がなければその旨を伝えて終了
   - `base <branch>`: 次を**両方**確認する。
     1. `git rev-parse --verify <branch>` — ブランチが存在すること
     2. `git status --porcelain` が空（作業ツリーがクリーン）であること —
        未コミット変更が残っていると、1 周目はブランチ間差分だけ・2 周目以降と
        `--mark` は作業ツリー全体が対象になり、依頼外の変更を修正候補にしたり、
        未レビューの内容をレビュー済みと記録したりする
     → クリーンでないときは **修正まで進めず、レビュー結果の報告だけを行う**
       （このとき Step 7 の `--mark` も実行しない）
   - `commit <sha>`: 次の 3 つを**すべて**確認する。
     1. `git rev-parse --verify "<sha>^{commit}"` — そのコミットが存在すること
     2. `git rev-parse HEAD` と一致すること — Step 5 の修正は「そのコミット」ではなく
        現在チェックアウト中の作業ツリーに当たるため、別リビジョンだと無関係なコードを壊す
     3. `git status --porcelain` が空（作業ツリーがクリーン）であること —
        未コミット変更が残っていると、1 周目はコミットだけ・2 周目以降と `--mark` は
        作業ツリー全体が対象になり、依頼外の変更を修正候補にしたり、
        未レビューの内容をレビュー済みと記録したりする
     → いずれかを満たさないときは **修正まで進めず、レビュー結果の報告だけを行う**
       （このとき Step 7 の `--mark` も実行しない）。修正が必要なら、対象コミットの
       チェックアウト / 作業ツリーをクリーンにする / `git worktree` の用意 の
       いずれかをユーザーに依頼する（勝手にチェックアウトや stash をしない）
2. **レビュー対象の fingerprint を控える**（Step 7 で使う。ラウンドごとに取り直す）:

   ```
   node "<このスキルのdir>/scripts/stop-hook.js" --print
   ```

   レビューは数分かかる。その間に作業ツリーが変わると（自分の追加編集、
   **同じツリーを共有する別セッション**）、Codex が読んでいない変更まで
   「レビュー済み」と記録してしまう。それを防ぐため、レビュー開始直前の状態を控える
3. **レビュー実行中は作業ツリーを触らない。** 待っている間に別の作業を進めない
4. 出力先はセッションの scratchpad ディレクトリにする（リポジトリ内に置かない）
5. カレントディレクトリを対象リポジトリのルートにして実行する:

```
node "<このスキルのdir>/scripts/run-review.js" --uncommitted --out "<scratchpad>/codex-review-round<N>.txt"
```

- 対象が `base` / `commit` 指定なら `--uncommitted` を `--base <branch>` /
  `--commit <sha>` に置き換える
- **必ずバックグラウンドで実行する。** レート制限に当たると設定の `wait_seconds` ぶん
  待ってから再実行するので、1 回の呼び出しが数十分に及びうる。
  呼び出し側のタイムアウトは
  `timeout_minutes × retry.max_attempts + sum(retry.wait_seconds)` 以上にする
- stdout の最終行が結果（`OK: ...` / `NG: ...`）。**指摘は `--out` のファイル**から読む
- `NG:` で終わったら**ループを中断**し、理由をユーザーに報告する（`--mark` もしない）
- **重要（実機確認済みの制約）**:
  - `--uncommitted` / `--base` / `--commit` はカスタムプロンプト `[PROMPT]` と併用できない
    （「cannot be used with '[PROMPT]'」エラーになる）。レビュー観点や見送りリストを
    Codex に指示することはできない
  - `--output-schema` は review モードでは無視され、結果は Codex 標準のテキスト形式で返る。
    構造化は Step 3 で Claude が行う
- 見送りリストの除外は Step 3 の Claude 側フィルタで行う
  （Codex が同じ指摘を再報告してくることは織り込み済み）

### Step 3: 結果をパースして判定する

`-o` のファイル（テキスト）を読む。おおむね次の形式で返る:

```
<総評（1〜数行）>

Review comment:

- [P1] <タイトル> — <ファイルパス>:<開始行>-<終了行>
  <本文（何が問題か・どう直すべきか）>
```

- `[P0]`（リリースブロッカー）〜`[P3]`（軽微）の優先度・タイトル・ファイル・行・本文を
  抽出し、指摘ごとに連番 id を振って内部リストにする（形式が多少揺れても趣旨で読み取る）。
  **`P0` を取りこぼさないこと** — 最も重大な指摘がここに来る
- 指摘の bullet が 1 件もない（総評のみ、「問題なし」相当）→ findings 0 件として Step 7 へ
- 出力は英語のことが多い。**ダイアログと報告では Claude が日本語に訳して提示する**
- **見送りリストとの突き合わせを必ず行う**。タイトル / ファイル / 指摘の趣旨を比較し、
  ユーザーが見送り済みの指摘と同趣旨のものは除外する（表現が変わっていても中身が
  同じなら同一とみなす）。残りがなければ Step 7 へ、あれば Step 4 へ

### Step 4: 対応する指摘を決める

**`unattended.enabled` が `true` のときはダイアログを出さない。**
「無人モード」（後述）の手順に従って自動で仕分けし、Step 5 へ進む。

以下は通常（有人）時の手順。AskUserQuestion で指摘ごとに選択させる。

**選択肢は 1 質問につき 2〜4 個でなければならない。** 1 個の質問はツール側で拒否され、
選択を集められずループが止まる。指摘の件数で組み立てを変える:

- **指摘が 1 件のとき**（最も起きやすいケース）: `multiSelect` を使わず、
  「対応する」/「見送る」の 2 択（単一選択）にする
- **指摘が 2 件以上のとき**: `multiSelect: true` で 1 質問につき 2〜4 指摘。
  5 件以上は質問を分けるが、**どのバッチも 1 件にならないように割る**
  （例: 5 件 → 3+2、9 件 → 3+3+3。4+1 や 8+1 は不可）。
  1 回の呼び出しで最大 4 質問。それ以上は複数回に分ける

共通:

- 質問文: 「どの指摘に対応する？（選ばなかったものは見送りとして記録するよ）」
- 選択肢のラベル: `[P0〜P3] #<id> <短いタイトル>`（日本語に訳し、短く保つ）。
  **重要度はラベルの先頭に必ず入れる** — description の中に埋めると、
  選択肢を一覧したときに軽重が分からない
- 選択肢の description: `<file>:<line> — <問題の説明> / 提案: <修正方針>`（日本語）
- 優先度の高い順（P0 → P3）に並べる
- 1 件のときの 2 択でも、質問文かラベルに重要度を必ず出す

選択された指摘 → **対応リスト**、選択されなかった指摘 → **見送りリスト**（以降のラウンドで除外）。

### Step 5: 対応リストを修正する

**このラウンドが `max_rounds` 回目なら、修正せずに Step 7 へ進む。**
ここで直すと作業ツリーが変わり、その変更を Codex は一度も読んでいないため、
Step 7 の `--mark` が拒否されて「もう一周」になる。つまり上限が上限として機能しない。
採用された指摘は **🔧 要対応（未レビュー）として報告に回す**。

- 対応リストの指摘だけを、最小の差分で修正する
- 指摘に便乗した無関係なリファクタリングをしない
- suggestion が既存設計と矛盾する場合は、設計に合う形に読み替えて修正し、その旨を報告に含める

### Step 6: 再レビュー（ループ）

Step 2 に戻る。ラウンド数をカウントし:

- **2 ラウンド目以降のスコープ**: Claude の修正は未コミットの作業ツリーに入るため、
  初回が `base` / `commit` 指定だった場合も **2 回目以降は `--uncommitted` に切り替えて**
  修正を含む状態をレビューさせる（切り替えた旨を完了報告に含める）。
  切り替えないと Codex は元のスナップショットを見続け、修正が一切検証されない
- 対応済みの指摘が再び報告された場合は「修正が不十分」とみなし、見送り扱いにせず
  再度ダイアログに出す
- 新規指摘 0 件（または全件が見送り済みと同趣旨）→ Step 7
- `max_rounds`（既定 5）に達したら打ち切り、未解決の指摘を報告に含める

### Step 7: レビュー済みとして記録し、完了報告する

1. **最後のレビューが `--uncommitted` だったなら必ず**、Step 2 で控えた
   fingerprint（**最後のラウンドのもの**）を渡して記録する:

   ```
   node "<このスキルのdir>/scripts/stop-hook.js" --mark <控えた fingerprint>
   ```

   - **記録されるのは「Codex が実際に読んだ差分」だけ。** レビュー中に作業ツリーが
     変わっていた場合、スクリプトは記録を拒否して終了コード 1 を返す。
     そのときは**現在の差分を対象に、もう一度 Step 2 から回す**
     （同じツリーを別セッションが触ったときにも起きる。詳細は `references/auto-loop.md`）
   - report-only で終わった `commit` / `base` モード（＝作業ツリーそのものを
     レビューしていない）では実行しない。未レビューの内容をレビュー済みと
     偽って記録することになり、自動ループが本来必要なレビューを飛ばしてしまう
   - 自動ループ（後述）を使っている場合、記録を忘れると Stop hook が次のターンでも
     発火し続ける。**指摘ゼロで終わったときも、max_rounds で打ち切ったときも、
     全件見送りだったときも、記録は必ず行う**
   - 自動ループを使っていない場合でも無害なので、条件を満たすなら常に実行してよい

2. **次の形式で締める。この形式はこのスキル専用。**

   利用者は複数セッションを並べて表示しており、**どれが終わったかを一目で判別する**
   のがこの形式の目的。**見出しと絵文字は毎回まったく同じ文字列を使う。**
   言い換えない、気を利かせて変えない、新しい絵文字を発明しない。
   ブレた時点でこの形式の意味が無くなる。

   ```
   ## ✅ codex-review 完了

   <何件指摘され、何を直したか。1〜2 文>

   ### 残タスク

   - ⚠️ 未検証 — 確認できていないこと と その理由
   - 🔧 要対応 — 未解決の指摘（max_rounds 打ち切りなど）
   - 💤 見送り — ユーザーが対応しないと判断した指摘（id / title / 重要度）
   - 🚫 保留 — 無人モードで実行しなかったもの（破壊的操作など）
   - ❓ 課題 — 無人モードで自分が判断した箇所（何を・なぜ・選ばなかった案）

   ### 📦 コミット

   <そのままコピペで実行できる git コマンド。詳細は下記>
   ```

   - **`## ✅ codex-review 完了` は完全固定。** この行だけは一字も変えない
   - 残タスクは**該当する区分の行だけ**残す。ひとつも無ければ `- なし` とだけ書く
   - 絵文字は上記の 6 種（✅⚠️🔧💤🚫❓📦）だけ。**それ以外は使わない**
   - **対応しなかった指摘は黙って落とさない。** 見送り・持ち越しとも必ず一覧に出す
   - 本文には往復回数・使用モデル・エフォート・レート制限での再試行回数も添える
   - **この形式を使うのはレビューのループを終えたときだけ。** 途中経過には使わない

### 📦 コミットの出し方

**`### 📦 コミット` は省略しない。** 「コミットは人間が行う」とだけ書いて終わらせず、
**そのままコピペで実行できるコマンドを必ず出す**（利用者が毎回自分で組み立てるのを避けるため）。

```bash
git add <対象> && git commit -m ":sparkles: <日本語の要約>"
```

- **メッセージはそのリポジトリの既存の慣習に合わせる。** `git log --oneline -10` を見て、
  gitmoji の有無・言語・粒度を揃える。**このスキルの好みを持ち込まない**
- 対象は**今回のレビュー範囲のファイル**。無関係な変更が混ざっているなら、
  `git add -A` ではなくファイルを列挙し、混ざっている旨も伝える
- **`git push` のコマンドは出さない。** push は人間が確認して実行する
- コミットすべきでない変更（生成物・一時ファイル・秘密情報の疑い）があれば、
  コマンドより先にそれを伝える

### 自動コミット（`auto_commit`、既定 `false`）

`~/.claude/codex-review.config.json` の `auto_commit` が `true` のときだけ、
上のコマンドを**実際に実行**し、結果（コミットハッシュ）を報告する。

- **`push` は絶対にしない。** `true` でも対象外
- **プロジェクトの規約が AI のコミットを禁じている場合は、`true` でも実行しない。**
  作業前に対象リポジトリの `CLAUDE.md` などを確認し、禁止されていれば指示だけ出して、
  そのことを報告に書く
- **無人モード中は `auto_commit` が `true` でもコミットしない。**
  レビューを人が確認していない状態でコミットが積み上がるのを避けるため、
  指示だけ残して週明けの判断に回す

## 初回セットアップ / 再設定（`/codex-review setup`）

**ここでも選択肢は 1 質問につき 2〜4 個の制約がかかる。** 候補が 4 個を超えるときは
絞って Other に逃がすこと（超えるとツール呼び出しが拒否され、セットアップが失敗する）。

1. `~/.codex/models_cache.json` を読む。`models` から `visibility == "list"` のものを
   `priority` 順に取り出す（slug / display_name / supported_reasoning_levels）
2. AskUserQuestion でモデルを選択させる:
   - 選択肢は上位 3 モデル ＋「Codex 本体の設定を継承」（= `mode: "inherit"`）の計 4 個
   - 一覧にないモデルは Other で slug を直接入力してもらう
3. `explicit` の場合、選択したモデルの `supported_reasoning_levels` から
   エフォートを選択させる（そのモデルが対応する値だけを選択肢に出す）。
   **エフォートが 5 段階以上あるモデルが実在する**（`low` / `medium` / `high` /
   `xhigh` / `max` / `ultra` の 6 段階など）。その場合は全部を並べず、
   **代表的な 3 個に絞り、残りは Other で入力してもらう**
4. `~/.claude/codex-review.config.json` に保存し、設定内容を復唱する
5. `models_cache.json` が存在しない・読めない場合は、モデル slug と
   エフォートを自由入力してもらう（値の検証は次回実行時のエラーで行う）

## モデルが廃止・変更されたとき

- `codex exec` がモデル不明・非対応エラーを返したら、ループを中断する
- `~/.codex/models_cache.json` を読み直して現行モデルを提示し、セットアップを再実行する
  （`~/.codex/config.toml` の `[notice.model_migrations]` に後継モデルの対応表が
  ある場合は、それを推奨として提示する）
- 設定を更新したらレビューを再開する

## 無人モード（`unattended.enabled: true`）

土日など人がいない間もレビューを回すためのモード。**ダイアログを出さず、重要度で
自動的に仕分ける。** 有人時の「人が採否を決める」設計から外れるので、
**設定で明示的に有効にしたときだけ**この経路に入る。

### 大原則: 無人時は判断を仰いで止まらない

**無人モードでは AskUserQuestion を一切使わない。** 人がいないので、聞いた時点で
週末がまるごと止まる。判断が必要になったら次のようにする。

1. **依頼の要件・既存の設計・このスキルの方針に照らして、最も近い選択肢を選んで進める**
2. 何をどう判断したか、**選ばなかった選択肢**とあわせて記録する（後述のファイル）
3. 完了報告で **「課題」として明示的に報告する**。勝手に決めた事実を埋もれさせない

これは指摘の採否に限らず、無人中に発生するすべての判断に適用する。

**ただし次は無人でも実行せず、保留して記録・報告に回す**（要件に近いかどうかに関わらず）:

- 破壊的な操作、データやファイルの削除、後戻りできない変更
- 依頼の要件を明らかに超える変更、無関係なリファクタリング
- コミット / プッシュ / 公開 / 外部への送信 / 課金を伴う操作
- ユーザーの既存の変更を捨てる操作

これらは「要件に近い判断」で押し切れる範囲を超えている。止まらないことより、
週明けに取り返しがつくことを優先する。

### Step 4 の代わりに行うこと

1. 指摘を重要度で二分する:
   - `unattended.auto_fix` に含まれる重要度（既定 `["P0", "P1"]`）→ **対応リスト**
   - それ以外（既定 `P2` / `P3`）→ **持ち越しリスト**。修正しない
2. 持ち越しと自動判断を、**次で得られるパス**のファイルに追記する
   （リポジトリ内には作らない。**パスを推測しないこと** — 名前が食い違うと、
   書いた内容が二度と見つからない）:

   ```
   node "<このスキルのdir>/scripts/stop-hook.js" --state-path
   ```

   ```json
   {
     "deferred": [
       { "severity": "P2", "title": "...", "file": "...", "line": 0,
         "description": "...", "suggestion": "...", "recorded_at": "<ISO8601>" }
     ],
     "decisions": [
       { "topic": "何について判断したか", "chose": "選んだ内容",
         "because": "要件のどこに照らして選んだか", "alternatives": ["選ばなかった案"],
         "recorded_at": "<ISO8601>" }
     ]
   }
   ```

   同趣旨の項目が既にあれば重複させない
3. **対応リストが空なら、そこでループを終える**（Step 6 に戻らず Step 7 へ）。
   自動修正の対象が無いということは作業ツリーが変わらないので、再レビューしても
   同じ指摘が返るだけ。`max_rounds` まで回すと、**無駄な API 呼び出しで
   レート制限を引き寄せる**（無人で放置する土日ほど効いてくる）
4. 対応リストがあれば Step 5 の手順で修正し、Step 6 でループする
5. ループ終了後、Step 7 で `--mark` する

### 有人に戻ったとき

`unattended.enabled` が `false` の通常実行では、Step 3 の後に
**持ち越しファイルを読む**。

- `deferred` の指摘を、今回の Codex の指摘と合わせてダイアログに出す。
  採否が決まった項目は `deferred` から消す
- `decisions` は**ダイアログではなく報告で見せる**（済んだ判断であって、選択肢ではない）。
  ユーザーが確認したと言うまで消さない
- 持ち越しが多いと選択肢の上限（1 質問 4 個）を超える。重要度順に、
  複数質問・複数回に分けて出す
- 「今回の Codex の指摘」と「持ち越し」が同趣旨なら 1 つにまとめる

### 注意

- **無人モードでも `--dangerously-*` 系フラグは使わない。** Codex は読むだけ
- 無人で自動修正する以上、**誤検知や方針違いの指摘を取り込むリスクがある**。
  週明けに必ず差分をレビューすること。コミットは人間が行う（このスキルはコミットしない）
- レート制限に当たったら `run-review.js` が待って再実行する。
  それでも `NG:` で終わったらループを中断し、状況を記録して報告に回す
  （待ち続けたり、叩き直したりしない）
- 週末が終わったら `unattended.enabled` を `false` に戻す

## 自動レビューループ（任意）

`/codex-review` を手で打つ代わりに、Stop hook で「Claude が応答を終えようとしたら
未レビューの差分を検知して自動でレビューを回す」運用もできる。
**利用者が自動ループを有効にしたい / 挙動を知りたいと言ったときだけ**
`references/auto-loop.md` を読んで案内する。

- 有効なリポジトリでは、ルートに `.codex-review-auto` が置かれている
- hook は判定するだけで、レビュー・ダイアログ・修正はこのスキルが担当する
- そのため **Step 7 の `--mark` を飛ばすと hook が発火し続ける**

## 注意事項

- Codex には書き込み・コマンド実行の権限緩和フラグを渡さない（レビューは読み取りのみ）
- レビュー結果ファイルや一時ファイルをリポジトリ内に作らない（scratchpad を使う）
- 認証エラー（未ログイン）の場合は、Codex 拡張または `codex login` でのログインを案内する
- このスキル自体はコミットを行わない
