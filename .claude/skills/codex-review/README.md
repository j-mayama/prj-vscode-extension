# codex-review

実装を伴うセッションを専用ブランチ＋worktreeへ分離し、実装完了時に Codex CLI が
独立レビューして、必要な修正と再レビューを行って結果を報告する。

```text
指示
  → 専用ブランチ＋worktree へ移動
  → 実装
  → Codex レビュー
  → P0 / P1 / P2 を修正 → 再レビュー
  → 指摘クリア
  → レビュー対象を自動コミット
  → 元のブランチへ merge
  → 報告（ブランチ / worktree / コミットハッシュ / merge 結果）
```

同じプロジェクトで複数のセッションを並行して動かしても、互いのファイル変更が
直接上書きされることはない。元のブランチへのmergeは1件ずつ実行し、同じ行の変更が
衝突した後続セッションは、自分のworktreeで競合を解消してCodex再レビュー・再コミット後に
mergeを再試行する。

merge 先は、**worktree を作ったときの元のブランチ**。作成時に記録しておき、merge 時に
照合する。そのため、作業中に元のチェックアウトで別のブランチへ切り替えていても、
そちらへ merge されることはない（記録と違えば merge せずに報告する）。

追跡済みの未コミット変更、別ブランチへの切り替え、進行中のGit操作など、自動処理で
触れると既存作業を壊す状況ではmergeせず、理由を報告に残す。並行セッション同士の通常の
競合は停止理由にせず、専用worktreeで解消・再レビューして統合まで続ける。

共有チェックアウトに未追跡ファイルだけがある場合は、その内容をコミットせずに保持して
専用worktreeへ分離する。merge先と同じパスでもローカル内容を未コミット変更として残す。

push はしない。worktree とブランチも自動では削除しない。

## 共有チェックアウトに未コミットの変更があるとき

worktree は HEAD から作られるため、追跡ファイルに未コミットの変更が残っていると分離できない
（そのまま分離すると、変更が共有側に取り残されて作業の出発点が変わる）。この場合は停止して、
次のどれにするかを聞く。`stash` / `reset` / `checkout` / コピーによる移送は行わない。

1. その変更を、いま入っているブランチへコミットしてから分離する
2. 自分でコミットしてから、あらためて指示し直す
3. このリポジトリでは分離しない（ルートに `.codex-review-no-worktree` を置く）

1 を選ぶと、まず対象の全件が表示される。同意したものだけがコミットされ、未追跡ファイルには
触らない（`git add -N` で追加予定にしたファイルも対象外）。表示のあとに変更が増えていた場合や、
**表示された同じファイルの中身が書き換わっていた場合**は、勝手に巻き込まずに中断して
もう一度確認する。表示から実行までの間に別のgit操作（`git add` など）が割り込んだ場合も、
どちらか一方だけが失敗するようになっていて、stageした内容が黙って消えることはない。

コミット後には取り消し用のコマンドも表示される。取り消せるのは、その WIP コミットが
ブランチの先端のままである間だけ。あとから別のコミットを積んだ場合は、そちらを巻き込まないよう
何もせずに止まる。

各セッションが編集前に worktree へ入るようになれば共有側はクリーンに保たれるので、
この確認が出るのはたいてい最初の一回だけ。

## 初期設定

利用するリポジトリのルートで、次を一度実行する。

```bash
node .claude/skills/codex-review/scripts/setup-auto.js --enable-schedule
```

## 指示方法

レビュー用のコマンドは不要。普段どおり実装を依頼すると、実装完了時に自動実行される。

```text
ログイン画面を実装して
```

未来の時刻から開始するよう指定した場合、その場では予約だけを登録する。指定時刻までは
Figma、MCP、リポジトリを解析せず、発火後に専用worktreeへ移動してから処理を始める。

現在の差分をレビューするだけの場合は、次のように指示する。

```text
実装せずに、現在の差分をレビューして
```

## レビュー対応

| 指摘 | 例 | 対応 |
| --- | --- | --- |
| P0 | アプリが起動しない、データが消える、セキュリティホールがある | 必ず修正 |
| P1 | 特定条件でクラッシュする、APIが正常に動かない、大きなメモリリークがある | 必ず修正 |
| P2 | エラーハンドリング不足、パフォーマンスを改善できる、重複コードがある | 必ず修正 |
| P3 | コメント、命名、フォーマット、任意のリファクタリング提案 | 修正せず結果にのみ表示 |

P0 / P1 / P2は勤務中・休日とも同じ基準で修正する。
勤務時間外・休日は質問で止めず、必要な判断を記録して進める。

## 結果

完了報告には次の内容が残る。

- 修正した指摘
- 結果にのみ表示するP3
- 修正失敗や安全上の理由で残ったP0 / P1 / P2
- 勤務時間外・休日に行った判断
- 作業したブランチとworktree
- 自動コミットの結果とコミットハッシュ
- merge の結果（merge できなかった場合はその理由）

自動コミットの対象は、セッション専用worktreeでレビューした全変更。実行中に生成した
未追跡ファイルも取り残さず、共有チェックアウトに元からある未追跡ファイルは含めない。

コミットは `git commit` ではなく `commit-tree` で組み立てる。`pre-commit` hook は
レビュー内容の検証後にファイルを書き換えて `git add` できるため、`git commit` を使うと
**レビューしていない内容がコミットへ入りうる**。そのぶん、リポジトリ側の
`pre-commit` / `commit-msg` / `post-commit` hook はこの自動コミットでは動かない。

merge はローカルのブランチまでで、push はしない。リモートへ出すかどうかは、完了報告の
コミットハッシュを見て判断する。
