# CLAUDE.md — VSCode 拡張機能 / Claude Code スキル 開発プロジェクト

開発を助けるツールを作って公開するリポジトリ（public）。成果物は 2 種類ある。

| 成果物                    | 置き場所               | 何か                                            |
| ------------------------- | ---------------------- | ----------------------------------------------- |
| **VSCode 拡張機能**       | `extensions/<name>/`   | VSCode 本体の機能を増やす。`.vsix` として配布   |
| **Claude Code スキル**    | `.claude/skills/<name>/` | Claude に定型作業の手順を持たせる。MD ＋ 付随物 |

このファイルには **常に効かせる方針だけ** を薄く置く。コピペ用テンプレートや各成果物の詳細は
`docs/` に分離し、**必要なときだけ読む**（コンテキスト節約のため）。

---

## 大原則：public リポジトリとして書く

このリポジトリは公開されている。**コミットするすべてのファイル（コード / ドキュメント / コメント）は、
文脈を知らない第三者が読んでも意味が通る内容にする。**

- **内部固有名詞を持ち込まない** — 他プロジェクト名・社内の呼び名・人名・社内 URL・チケット番号など、
  外部の人に伝わらない・晒すべきでない情報は書かない（例：別リポジトリの名前をそのまま参照しない）
- **「〜から流用」の痕跡を残さない** — 由来を書く必要があるときは、汎用的な一般論として書き直す
- 迷ったら「初見の OSS 利用者が読んで理解できるか？」で判断する
- 秘密情報の扱いは後述の「セキュリティ」に従う

**スキルは特に危ない。** スキルは実案件のリポジトリで育ててからここへ持ち込む流れになりがちで、
その過程で客先名・実ディレクトリ名・テーブル接頭子・認証情報が「例示」として自然に紛れ込む。
拡張機能のコードより混入しやすいので、**持ち込みを汚染の入口とみなして必ず汎用化を通す**
（→「スキルを作る / 持ち込むとき」）。

### ファイルの役割分担

- **README.md = 人間（利用者・第三者）向け** — Claude のコンテキストに載せる必要のない情報を置く。
  基本は **プロジェクト概要** と **各成果物の説明（何ができるか / インストール・使い方）** だけ。
  開発フローや作業方針・テンプレートは README に書かない
- **CLAUDE.md = Claude 向けの作業方針** — 開発の進め方・ルール。README の内容と重複させない
- **SKILL.md = そのスキルを実行する Claude 向けの手順書** — 読者が README と違うので重複させない。
  「このリポジトリの開発方針」ではなく「そのスキルの実行手順」だけを書く
- 成果物が増えたら README には「一覧＋概要」を並べ、詳細な設計・調査記録は
  `docs/extensions/<name>.md` / `docs/skills/<name>.md` に置く

## 開発フロー：拡張機能を作るとき（この順序を必ず守る）

新しい拡張を作るときは 2 フェーズを順番に通す。**フェーズ1を飛ばさない。**

### フェーズ1：事前調査（作る前に必ず）
1. Marketplace（`https://marketplace.visualstudio.com/`）と Open VSX（`https://open-vsx.org/`）で近い拡張を検索。
   WebSearch で「VSCode extension + 目的（英語）」が速い
2. 調査結果を報告する（拡張名 / 作者 / インストール数 / 最終更新 / できること / 不足点）
3. 作るか検討 — 既存で足りるなら **作らない提案** をする。新規 / フォークする場合は
   **ユーザーの判断を仰いでから** フェーズ2へ進む（勝手に実装を始めない）
   - 報告フォーマットは `docs/extensions/_TEMPLATE.md`

### フェーズ2：実装（作ると決まってから）
1. 使う API / Contribution Point のベストプラクティスを公式ドキュメント（`https://code.visualstudio.com/api`）で調べる
   （バージョンで API が変わる。断言できないときは「確認できない」と言う）
2. 調べた内容を `docs/extensions/<name>.md`（テンプレートは `_TEMPLATE.md`）に記載してから実装する
3. 実装 → F5 デバッグ → テスト

## スキルを作る / 持ち込むとき

### 持ち込み時の汎用化（例外なし）

実案件で育てたスキルをここへ入れるときは、**ステージする前に**次を通す。

1. **客先・社内に固有の値をすべて置換する** — 客先名 / ステージング名 / 実ディレクトリ名 /
   テーブル接頭子 / 計測タグの実値。プレースホルダ（`[プロジェクト名]`）か
   汎用例（`example` / `my-project`）にする
2. **外部の社内文書に依存した記述を書き直す** — 「手順書の③」「Notion の①〜⑭」のような参照は
   第三者には解決できない。**内容そのもの**を書く
3. **テンプレート内の認証情報欄に実値を書かせない** — 「ユーザー名 / パスワード」を持つ
   テンプレートは、実値ではなく *管理場所を参照させる* 指示にする。
   テンプレが悪い手本になると、生成される全プロジェクトに伝播する
4. **`/secret-scan` を実行する** — 汎用スキャナは客先名・社内固有名詞を検出しないので、
   自動ツールの結果が 0 件でも上記 1〜3 は人が読んで判断する

### スキルの体裁

- `SKILL.md` の frontmatter に `name` / `description` / `version` を置く
- `description` は **「いつ発動すべきか」が一読でわかるように**書く（Claude はここだけ見て読むか判断する）
- 更新時は `version` を上げ、`## 変更履歴` に日付と変更内容を追記する
- **決定論的にやるべき処理は LLM に書かせない** — ファイル生成・値の置換は同梱スクリプトに寄せる
  （毎回同じ結果になるべきものを推論に任せると静かに壊れる）

### 公開前ゲート：スキルの独立レビュー（必須）

スキルの追加・更新後は、実装と公開可否レビューを別フェーズとして扱う。
**実装者の説明・変更履歴・「汎用化済み」という申告を信用せず、作業ツリーを正としてゼロから検証する。**
可能なら実装時とは別のコンテキストまたは別エージェントでレビューする。

#### 1. 対象確定と全件通読

1. `git ls-files -co --exclude-standard -- .claude/skills/` で、未追跡を含むコミット候補を列挙する
2. `SKILL.md` / `scripts/` / `templates/` / `references/` の対象ファイルをすべて直接読む
3. grep・secretlint・gitleaks は補助に限り、通読の代わりにしない
4. 「例」「サンプル」「参考」「既定値」の中も実案件由来を疑う
5. 既知の禁止語にヒットしなくても、初見の OSS 利用者に説明できない識別子は汎用化する

#### 2. 危険操作の棚卸し

次を含む処理をすべて列挙し、発動条件・対象範囲・失敗時の影響を確認する。

- ファイルの作成・上書き・削除、ディレクトリの再帰削除
- `rm` / `rmtree` / `Remove-Item` / `docker rm` / `docker network rm`
- `sed`・正規表現・テンプレート展開による設定値の置換
- `--force`、既存出力先の削除、DB情報・テーブル接頭子の更新
- ホストディレクトリ・SSH鍵・ホームディレクトリの mount
- ホストポート公開、外部URLへの送信・ダウンロード

危険操作は次をすべて満たさなければ公開不可とする。

1. 既存ファイルはデフォルトで変更しない
2. 破壊的操作は明示オプションまたはユーザー確認を必要とする
3. 削除対象は絶対パス・所有ラベル・作業ディレクトリ等で現在のプロジェクトに限定する
4. 対象が不明・異常な場合は削除せずエラー終了する
5. 単純なローカル認証を使うサービスは `127.0.0.1` にのみ公開する
6. 失敗時に既存ファイルを中途半端に変更しない

#### 3. 文書と実装の相互照合

`SKILL.md` の「上書きしない」「既存値を保持」「現在のプロジェクトだけ」「localhost 限定」
「任意」「明示された場合のみ」「安全」という記述は、通常経路から実際に呼ばれる関数・コマンドまで追跡する。
**文書に書かれているだけでは合格にしない。** 実装と一致しない場合はコードと文書の両方を直す。

#### 4. 隔離環境での実行テスト

ファイル生成・設定変更を行うスキルは、一時ディレクトリで最低限次を実行する。

1. 初期状態からの生成
2. 既存ファイルがある状態での通常実行（sentinel 値が不変であることを前後比較）
3. 明示的な修復・上書きモード
4. 既定値以外のディレクトリ名・接頭子・出力先
5. 同じコマンドの2回実行（冪等性）
6. 異常なパス・不完全な入力で、既存データを消さず安全に失敗すること

終了コード `0` だけで合格にしない。実行前後のファイル内容と生成物を比較する。
テンプレートは参考資料ではなく、利用者環境へコピーされ実行される製品として扱い、
プレースホルダ置換、参照先、URL・ポート・プロトコル、構文を生成後のファイルで検証する。

#### 5. 公開可否の完了条件

次の証拠を提示できるまで「公開して問題ない」と報告してはならない。

- 読んだファイル数と対象範囲
- 危険操作の一覧と、安全だと判断した根拠
- 実行した検証コマンドと結果
- 通常実行で既存ファイルが保持された比較結果
- 非既定値で生成できた結果
- secretlint 等の自動検査結果
- frontmatter の `version` と最新変更履歴の一致
- 未確認・未実行項目

検証できなかった項目は「問題なし」ではなく **「未確認」** と報告する。
自動ツールが 0 件でも、通読・危険操作の確認・隔離実行を省略しない。

## 標準スタック（拡張機能）

特別な理由がない限りこの構成に統一する。

- TypeScript（`strict: true`）/ バンドラーは esbuild
- テスト：`@vscode/test-cli` + `@vscode/test-electron` / Lint：ESLint
- 公開：`@vscode/vsce`（Marketplace）・`ovsx`（Open VSX）
- `engines.vscode` と `@types/vscode` のバージョンを一致させる
- スキャフォールド：`npx --package yo --package generator-code -- yo code`

## 実装の基本方針（拡張機能）

- **アクティベーションは遅延** — `activationEvents` は最小限（1.74+ は `onCommand` 等を contribution から自動生成）
- **Contribution Points 優先** — できることは `package.json` の宣言で。命令的コードは最後の手段
- **UX ガイドライン準拠** — `https://code.visualstudio.com/api/ux-guidelines/overview`
- **リソースは dispose** — `context.subscriptions.push(...)` に登録
- **設定は `configuration` contribution で** — ハードコードしない
- **エラーを握りつぶさない** — `window.showErrorMessage` 等でユーザーに見せる
- **困ったらまず Web で最新のベストプラクティスを調べる** — 手元の知識で書く前に公式・主流手法・既知の落とし穴を確認し、「採用案 / 不採用案 / 理由」を提示してから実装する

## セキュリティ（例外なく守る）

- **git の commit / push は必ず人間（開発者）が行う。AI は勝手に commit / push しない。**
  一度でも機密情報が混入すると履歴に残り、削除が困難になるため。AI は変更内容を提示するまでに留め、
  最終的にステージ・コミット・プッシュするのは人間。ユーザーが明示的に頼んだ場合のみ commit まで行い、**push は必ず人間が確認して実行する**
- シークレット（API キー / トークン / パスワード）を **平文でコミットしない**。`.gitignore` で
  `.env*` / `*.pem` / `*.key` / `.vscode/settings.json` / `.claude/settings.local.json` を除外
- **値の生成・保管は開発者（人間）に依頼** — `.env.example` にはキー名だけ書く
- **公開前に `/secret-scan`**（`.claude/skills/secret-scan/`）— シークレット・PII・内部固有名詞・
  `.vsix` 梱包物を検査する。**自動ツールが 0 件でも安心しない** — secretlint / gitleaks は
  客先名や社内固有名詞を知らないので、そこは人が読んで判断する
- **見つけた認証情報は「消す」より先に「無効化（ローテート）」** — ファイルから消しても、
  既にプッシュ済みなら流出は取り消せない
- **ユーザー認証情報は `vscode.SecretStorage`** に保存（`settings.json` / `globalState` に平文で置かない）
- **公開前に `npx @vscode/vsce ls`** で `.vsix` 梱包内容を確認。`.vscodeignore` で `src/` 生ソース・`.env`・テストを除外。公開 PAT はローカルのみ（CI では暗号化 Secret）
- **Webview は CSP + `nonce` 必須**。`enableScripts` は必要時のみ、`localResourceRoots` で限定、外部データは必ずエスケープ（`innerHTML` 直挿し禁止）
- **依存追加時は `npm audit`**。ワークスペース内容の無断外部送信禁止（テレメトリはオプトイン）

## ディレクトリ構成（目安）

```
├── CLAUDE.md                 ← 常に効かせる方針（薄く保つ）
├── README.md                 ← 人間向け：成果物の一覧と使い方
├── docs/extensions/          ← _TEMPLATE.md ＋ 拡張ごとの調査・実装記録
├── docs/skills/              ← スキルごとの設計・調査記録（必要になったら）
├── extensions/<name>/        ← 拡張ひとつ ＝ 独立パッケージ
│   ├── src/extension.ts      ← エントリポイント（activate / deactivate）
│   ├── package.json          ← マニフェスト（contributes / activationEvents）
│   ├── esbuild.js / tsconfig.json
│   └── .vscodeignore         ← .vsix 梱包から外すもの（.gitignore とは別物）
├── .claude/skills/<name>/    ← スキルひとつ ＝ SKILL.md ＋ 付随物
│   ├── SKILL.md              ← frontmatter（name / description / version）＋ 手順
│   ├── scripts/              ← 決定論的にやる処理（LLM に書かせない部分）
│   ├── templates/            ← 展開元テンプレート
│   └── references/           ← 参照専用の資料（必要なときだけ読ませる）
└── .vscode/                  ← launch.json（F5）/ tasks.json
```

## よく使うコマンド

```bash
# 共通
/secret-scan           # 公開前の機密チェック（コミット前に必須）

# 拡張機能
npx --package yo --package generator-code -- yo code   # 新規スキャフォールド
npm run watch          # 開発（watch ビルド）。デバッグは VSCode で F5
npm test               # テスト
npx @vscode/vsce ls    # .vsix 梱包内容の確認（publish 前に必須）
npx @vscode/vsce package / publish   # パッケージング / Marketplace 公開
npx ovsx publish       # Open VSX 公開
```
